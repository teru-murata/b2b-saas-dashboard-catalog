# Evidence Model

| category | what the catalog should show | why it matters to BtoB SaaS teams | demo evidence | quality evidence | pitch note |
|---|---|---|---|---|---|
| data-rich dashboard UI | Summary cards, dense sections, filters, and clear hierarchy. | Teams need complex data to become decisions quickly. | Overview dashboard with sample metrics and state handling. | Responsive checks and content hierarchy review. | Lead with how the screen reduces analysis time. |
| data table interaction | Sorting, filtering, status labels, empty states, and readable density. | Many SaaS workflows are table-first and operational. | Static sample table with realistic business fields. | Unit tests for transforms and keyboard review. | Explain tradeoffs between density and comprehension. |
| analytics visualization | Simple charts that support a decision rather than decoration. | Visual trends help users compare periods and spot exceptions. | A later chart panel tied to dashboard data. | Chart labels, fallback text, and visual review. | Keep charts legible and directly connected to the table. |
| reusable component architecture | Shared primitives for cards, tables, badges, filters, and states. | Product teams need consistent changes without rewriting screens. | Components reused across dashboard sections. | Type checks and variant usage review. | Emphasize maintainability over visual novelty. |
| typed data boundary | Explicit TypeScript types for sample data and UI props. | Typed contracts reduce integration ambiguity and review risk. | Static data model consumed by typed components. | Typecheck and focused unit coverage. | Show how UI assumptions are encoded. |
| design-system discipline | Tokens, spacing, focus states, and predictable variants. | Consistency helps designers and engineers collaborate. | Token-backed UI elements used in real sections. | Contrast and focus-state review. | Keep design rules practical and visible. |
| accessibility | Semantic structure, keyboard access, focus visibility, and readable contrast. | Enterprise software must support diverse users and review standards. | Accessible headings, buttons, tables, and status text. | Manual checklist and later browser smoke checks. | Present accessibility as engineering quality, not polish. |
| performance | Render strategy, limited client work, and data-volume rationale. | Dense dashboards can become slow without clear boundaries. | Notes near table and dashboard sections. | Basic render observations; deeper measurement later. | Be honest about what is measured versus planned. |
| testing | Focused checks for data transforms, rendering, and navigation. | Tests make remote review and iteration safer. | One focused unit test and later smoke path. | Local check command summary. | Small real tests beat broad test plans. |
| documentation | Implementation notes, assumptions, and decision records. | Async teams need context without meetings. | Delivery notes and implementation notes sections. | Review for clarity and stale claims. | Documentation should explain decisions, not inflate scope. |
| GitHub delivery workflow | Issue-style breakdown, commit hygiene, and PR-ready checklist. | Remote teams evaluate how work arrives for review. | Delivery notes with checklist and assumptions. | Local commit and check summaries. | Show review readiness without implying prior team history. |
